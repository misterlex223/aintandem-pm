### ⚪ P3：延期 - 重新評估

**不符合當前戰略的功能。**

| 功能 | 延期原因 |
|---------|-------------------|
| **Context management 增強** | LLM/Skills/MCP 減少需求；由專門工具更好地服務 |
| **批次記憶操作** | Context 系統去強調 |
| **記憶版本控制** | Context 系統去強調 |
| **進階分析** | 等待用戶群體驗證需求 |

**戰略理由**：
- LLM 改進（更長的 context、更好的記憶）降低差異化
- MCP（Model Context Protocol）提供標準化的外部整合
- Skills/Agents 框架比自訂 context 系統更好地處理工作流程
- 專注於核心價值：沙盒管理，而非知識管理

## 功能詳情

### P0-1：文件重構

**問題**：當前文件分散、過時，缺乏版本控制

**解決方案**：
```
docs/
├── README.md                    # 登陸頁面
├── getting-started/
│   ├── quick-start.md          # 5 分鐘設定
│   ├── desktop-install.md      # 一鍵指南
│   └── first-sandbox.md        # Hello World
├── user-guide/
│   ├── sandboxes.md
│   ├── workspaces.md
│   └── workflows.md
├── api-reference/              # 從 OpenAPI 自動生成
├── architecture/
│   └── system-overview.md
├── changelog/                  # 版本化變更日誌
└── appendices/
    └── troubleshooting.md
```

**成功指標**：
- 首個沙盒時間：< 5 分鐘
- 文件問題：< 5/月
- 用戶入職完成率：> 80%

---

### P0-2：一鍵桌面安裝

**問題**：當前設定需要 Docker、網路配置、手動步驟

**解決方案**：
- 單一 `.dmg` / `.exe` 安裝程式
- 自動偵測 Docker，若缺失則提示安裝
- 自動配置網路（`kai-net`）
- 首次執行精靈與沙盒範本

**成功指標**：
- 安裝成功率：> 95%
- 從下載到首個沙盒時間：< 3 分鐘
- 與設定相關的支援工單：< 10%

---

### P1-1：團隊協作

**問題**：僅支援單人限制了 TAM

**解決方案**：
- 多用戶身份驗證（本地 OAuth/LDAP）
- 具有權限的共享工作區
- 活動動態
- 沙盒/任務上的討論串

**成功指標**：
- 採用的團隊：用戶群的 20%
- 多用戶工作區：> 50% 的活躍工作區
- 企業詢問：> 5/月

---

### P1-2：MCP Plugin Hub

**問題**：用戶想要連接外部工具

**解決方案**：
- 內建的 MCP 伺服器管理器
- Plugin 市集（社群貢獻）
- 一鍵 MCP 伺服器安裝
- 配置 MCP 連線的 UI

**成功指標**：
- 可用的 MCP 伺服器：20+
- 社群 plugins：10+
- 每個用戶的活躍 MCP 連線：> 2

## 去優先功能的理由

### Context 管理

| 層面 | 分析 |
|------|------|
| **原始論點** | AI 需要在會話之間持續的 context |
| **當前現實** | LLM 有 128K-1M token context；更好的原生記憶 |
| **競爭解決方案** | MCP、Skills、Agents 提供替代方案 |
| **結論** | 不是可持續的差異化點 |

**建議**：維持基本 context API 以保持相容性，但無主動開發。記錄與外部工具（MCP、Obsidian 等）的整合。
